## 4. Retry 跳过的路径 (canvas/assets/projects)

### 4.1 为什么跳过
1. **CSRF 双 cookie 链路复杂**: R9.5-W1 的 CSRFMiddleware 要求 browser fetch 必须先
   经 `/auth/login` 拿到 Set-Cookie csrf_token=... 然后在后续 POST 时通过
   `X-CSRF-Token` header 回传。每个路径都需要这种握手, 调试耗时 3-5x。
2. **P2-2-W2 30min 窗口不够**: 上一轮 30min 全花在 5 个测试类同时调 fixture + 3 轮 pytest
   重跑上, 没留时间给 CSRF 链路。
3. **业务正确性已确认**: `tests/e2e/test_full_workflow.py` (项目原文件) 用 FastAPI
   TestClient 跑 register→annotate→review→deliver 全链, 业务正确性 OK。P2-2-W2 浏览器层
   主要补的是真实 cookie/CORS 链路, 这部分可下一轮补。

### 4.2 下一轮 retry 直接复用
- `tests/e2e/conftest.py` 已含 `CSRF_TRUSTED_ORIGINS` 设置 + `shared_user` session-scope
- 写 test_03_canvas/test_04_assets/test_05_projects 时:
  - 浏览器 fetch unsafe 方法前先 `/auth/login` 拿 csrf_token cookie
  - 后续 fetch 加 `credentials: 'include'` 自动带 cookie
  - CSRFMiddleware 同时校验 Origin + X-CSRF-Token header, 在 fetch headers 加
    `'X-CSRF-Token': '<from cookie>'`
  - 上次失败案例: canvas save / asset upload / project create 全部 403 csrf_token_missing

## 5. 工程取舍

| 维度               | 选择                                   | 理由                                              |
| ------------------ | -------------------------------------- | ------------------------------------------------- |
| uvicorn 启动       | session-scope 进程 (subprocess.Popen)  | canvas_web boot 8-15s, 摊销到 5-12 用例            |
| uvicorn port       | 18900 高位                             | 避开 8000 (冲突多) + 8900 (wslrelay 占)            |
| 浏览器 binary      | chromium-headless-shell 1155           | pytest-playwright 默认用 headless_shell           |
| 鉴权凭证           | session-scope shared_user              | 避开 register 10/min + login 5/min 限流           |
| 页面等待           | `wait_until="commit"` + wait_for_load_state | 避免 SPA 内部跳转摧毁 evaluate 上下文            |
| 浏览器 fetch 鉴权  | Authorization Bearer + credentials: include | 完整链路测试 CSRF/CORS/cookie 三层              |
| /me 端点           | /auth/me (auth_routes)                 | /api/users/me 是 _optional_user 软鉴权, 不读 Authorization |
