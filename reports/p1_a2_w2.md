## 1. 任务背景

R2/R3 已有 `webhook_routes.py` (~580 行) 提供基础 CRUD, 但缺:
1. **重试队列** — 数据库有 `max_retries` 字段, 但没有任何重试实现
2. **死信队列 (DLQ)** — 重试耗尽后无归处
3. **事件类型不足** — 仅 19 个, 任务要求 ≥30
4. **订阅隔离** — 无 user_id / tenant_id 字段
5. **secret 强度校验** — 仅检查长度 ≤256, 不要求 ≥32
6. **rotate-secret 端点** — 缺失
7. **架构分层** — 所有逻辑在 routes 里, 没有 engines/ 层的 WebhookEngine 类

本任务把这 7 项缺口全部补齐。

## 4. 退避算法

**Schedule**: `[1, 4, 16, 60, 300]` 秒 (5 次)

**触发链**:
```
attempt 1 fail → next_retry_at = now + 1s
attempt 2 fail → next_retry_at = now + 4s
attempt 3 fail → next_retry_at = now + 16s
attempt 4 fail → next_retry_at = now + 60s
attempt 5 fail → DLQ
```

**Worker**: 调用 `engine.process_retry_queue()` (或挂后台 asyncio 定时器)
- 扫描 `next_retry_at <= now` 的所有条目
- 删除扫描到的条目 (避免重复处理)
- 对每条: 查订阅, 若仍 active → 重新走 `_deliver_one(attempt=N+1)`
- 若订阅已删/停用 → 直接转 DLQ (orphan 处理, 见 test_24)

**总最坏延迟**: 1 + 4 + 16 + 60 + 300 = **381 秒** (从首次失败到最后入 DLQ)

## 5. HMAC 签名规范

**算法**: HMAC-SHA256
**Header**: `X-IMDF-Signature: sha256=<hex>`

**Body 覆盖范围**: 完整 envelope JSON
```json
{
  "event_type": "asset.created",
  "delivery_id": "del_abc123",
  "webhook_id": "wh_xyz789",
  "timestamp": "2026-06-22T01:00:00+00:00",
  "attempt": 1,
  "max_attempts": 5,
  "data": { ... 业务字段 ... }
}
```

**接收端校验伪代码** (测试 6 验证):
```python
def verify(payload, secret, signature):
    expected = hmac.new(secret.encode(), payload, sha256).hexdigest()
    sig = signature.split("=", 1)[-1] if signature.startswith("sha256=") else signature
    return hmac.compare_digest(expected, sig.lower())
```

**失败语义**: 签名不匹配 → 接收端应返回 401/403。生产 IMDF 端在 `_deliver_one` 里捕获为 `http_status=401` → 进入重试。

## 7. 安全

| 风险 | 防御 |
|------|------|
| SSRF (Webhook URL 指向内网) | `validate_webhook_url` (R2 既有) — 拒 localhost/private/link-local IP, DNS 解析后二次校验 |
| 弱 secret | `_validate_secret` — 必须 ≥32 字符, 无空白, 否则 400 |
| 跨用户越权 | `dispatch(user_id=...)` + `list_subscriptions(user_id=...)` — 按 owner 过滤; API 默认从 `X-User-ID` header 取 |
| 重放攻击 | 接收端应校验 `timestamp` 在合理窗口 (e.g. ±5min); 投递唯一性 `delivery_id` 存于 body + DB |
| Secret 泄露 | 列表/详情接口 mask secret (只显示前 8 后 4), 只在创建/轮换时一次性返回明文 |
| 常时间签名比较 | `_verify_signature` 用 `hmac.compare_digest` 防时序攻击 |

## 9. 已知限制

1. **没有挂后台定时器**: `process_retry_queue()` 是同步方法, 需生产部署在 FastAPI `startup` 事件里 `asyncio.create_task` 每秒扫一次。
2. **DLQ 自动归档**: 当前只有手动重投。后续可加 cron: DLQ entry 超 7 天自动归档到 `dlq_archive` 表。
3. **HTTP 注入点**: `_http_post_fn` 默认 aiohttp 真实发送。生产可注入带 circuit breaker / connection pool 的实现。
4. **事件源没广播**: `dispatch()` 只走 webhook 表, 没同时 publish 到 NSQ/Kafka。后续可在 `dispatch()` 入口加 publish。
5. **跨租户严格隔离**: `tenant_id` 字段已存, 但 dispatch 没强制要求 (空 tenant_id 视为全局)。生产多租户环境需要在 middleware 层强制 tenant_id。
