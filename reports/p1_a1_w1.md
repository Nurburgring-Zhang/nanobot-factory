## 4. C2PAEngine design

**Class: `C2PAEngine(cert_path, key_path, issuer, ttl_seconds)`**

| Field / method | Behavior |
|---|---|
| `_load_or_generate_keys()` | Loads existing PEM cert+key if both exist; otherwise generates a self-signed X.509 cert with RSA-2048 key, 10-year validity, BasicConstraints + KeyUsage(digital_signature, content_commitment) + SubjectKeyIdentifier. |
| `cert_fingerprint()` | SHA-256 hex of DER cert — used in manifest binding. |
| `sign_asset(asset_path, claim)` | 1) `SHA-256` of asset file, 2) build manifest with claim_generator + actions + hash chain link to previous manifest, 3) compute `manifest_hash = SHA-256(canonical_body)`, 4) sign `prev_hash ‖ manifest_hash` with RSA-PSS-SHA256, 5) write sidecar `<asset>.c2pa.json`. |
| `verify_signature(asset_path)` | Recomputes asset hash, manifest hash, then verifies RSA-PSS signature with cert's public key. Returns `(is_valid, manifest_dict)` with `reason` populated. |
| `revoke(manifest_id)` | Appends to `self.crl`; flags in-memory chain; rewrites matching sidecar JSON files. Idempotent. |
| `get_crl()` | List of `{manifest_id, revoked_at}`. |
| `get_manifest(id)` / `list_manifests()` | Chain lookups. |

**Hash chain**: each manifest stores `previous_manifest_id` + `previous_manifest_hash`
(chain tip). Signature input = `prev_hash ‖ manifest_hash` (binary concat, UTF-8 of
hex), exactly mirroring how Bitcoin / RFC 3161 timestamp chains bind to the prior
block.

**Canonical form**: RFC 8785-lite — sort keys, separators `(',', ':')`, UTF-8.
The canonical body covers every manifest field except `signature` and
`manifest_hash`; `manifest_hash` is then computed over that canonical body and
included before signing.

**Sidecar format**: `<asset_path>.c2pa.json` — forward-compatible with future
binary asset embedding (JPEG APP1 / MP4 UUID box). Real C2PA embeds CBOR in
those segments, but for IMDF's JSON-RPC API we keep a parseable sidecar.

## 5. API endpoints (all under `/api/v1/copyright/c2pa/*`)

| Method | Path | Status codes | Description |
|---|---|---|---|
| POST | `/c2pa/sign` | 200 / 400 / 404 / 500 | Sign asset; persist manifest; return `manifest_id` + manifest. |
| GET  | `/c2pa/verify/{asset_id}` | 200 / 404 | Verify; `asset_id` is either `manifest_*` (lookup in DB) or an asset path. |
| GET  | `/c2pa/manifest/{manifest_id}` | 200 / 404 | Retrieve stored manifest JSON. |
| POST | `/c2pa/revoke/{manifest_id}` | 200 / 404 / 409 | Add to CRL; 409 on already-revoked. |
| GET  | `/c2pa/crl` | 200 | List revoked manifests. |

**Schema additions**:
- New SQLite table `c2pa_manifests` (id, asset_path, asset_hash, claim_json, manifest_json, cert_fingerprint, revoked, created_at) + index on asset_path.
- Persisted CRL is restored into the engine singleton on first request after process start.
- New env vars: `C2PA_CERT_PATH` (default `data/c2pa_cert.pem`), `C2PA_KEY_PATH` (default `data/c2pa_key.pem`).

## 8. Production notes / follow-ups

- **Key persistence**: key is saved unencrypted at `data/c2pa_key.pem` with
  `chmod 0o600` (best-effort on Windows). Production must wrap with KMS / HSM
  and password-protect the private key (use `BestAvailableEncryption`).
- **Sidecar vs in-band**: C2PA real spec embeds the manifest as CBOR in JPEG
  APP1 / MP4 UUID boxes. The sidecar `.c2pa.json` approach keeps the API simple
  and works for any binary format; future work can add a `embed_into_image()`
  helper using Pillow EXIF for JPEG/PNG.
- **Trust chain**: the self-signed cert is for development. Production needs a
  CA-issued cert (e.g. C2PA Trust List members) so that third-party verifiers
  can chain trust.
- **Clock skew**: `not_valid_before` is set to `now - 5min` to tolerate small
  clock drift; `not_valid_after` is 10 years. Manifests use 5-year TTL.
- **Concurrency**: singleton engine is not thread-safe for signing (uses
  in-memory chain). For multi-worker deployments, consider Redis-backed chain
  or use the DB-stored manifest_chain instead of in-memory list. Current usage
  is single-process (uvicorn workers=1 by default).
- **CRL distribution**: a real CRL is X.509 CRL signed by the CA. Our in-memory
  list is suitable for the IMDF platform's internal use; for interop, a future
  step would be to publish a signed CRL endpoint at `/.well-known/c2pa-crl`.
