## 2. 验证器契约 (R2 阶段已交付, R2.5-W4 复用)

### 2.1 `validate_cron(expr, name="cron")`
- **职责**: 校验 5 字段 cron 表达式 (`分 时 日 月 周`)
- **支持语法**: `*` 通配 / `a,b,c` 列表 / `a-b` 范围 / `*/N` 步长 / `a-b/N` 范围步长
- **拒绝**: 字段数 ≠ 5 / 越界 (minute ≥ 60, hour ≥ 24 等) / 非法 token
- **错误**: HTTPException(400), 中文 + 含字段名

### 2.2 `validate_webhook_url(url, name="url")`
- **职责**: 校验 Webhook URL 合法 + 阻断 SSRF
- **3 层防御**:
  1. 字面 hostname 黑名单 (`localhost` / `metadata.google.internal` / `169.254.169.254`)
  2. 字面 IP 私网检查 (`ipaddress.is_private` / `is_loopback` / `is_link_local` / `is_reserved` / `is_multicast` / `is_unspecified`)
  3. DNS 解析后私网检查 (resolve → IP → 二次私网判断)
- **拒绝**: 非 http(s) scheme / 超 2048 字符 / 私网 IP / 内网 hostname

### 2.3 `validate_task_id(value, name="task_id")`
- **职责**: 校验异步任务 ID 严格命名空间
- **7 命名空间**: `task_` / `job_` / `batch_` / `run_` / `del_` / `wh_` / `mig_`
- **主体**: 4-64 位 `[a-zA-Z0-9_]`
- **拒绝**: 空 / 越界 / 非法字符 / 未知前缀
- **注**: 通用资源 ID (含 `preset_xxx`) 仍走 R1 `validate_id`, 详见 `task_id_validator.py` 注释

### 2.4 `SchedulerHistoryParams` (Pydantic v2 BaseModel)
- **职责**: 调度历史查询 — 分页 + job_id + 时间范围 + 状态
- **继承**: `PaginationParams` (通用分页, MAX_LIMIT=200)
- **额外字段**:
  - `job_id`: `^[a-zA-Z0-9_\-]{1,128}$` (兼容 `preset_xxx`)
  - `start` / `end`: `date`
  - `status`: `Literal["success", "failed", "running", "pending"]`
- **校验**:
  - `start > end` → HTTPException(400) (model_validator, raise HTTPException 而非 ValueError, 避免 FastAPI 500)
  - 跨度 > 365 天 → 400

## 3. 引擎层修复 (R2.5-W4 新增)

R2 阶段把 4 个验证器接入路由, 但路由调用引擎时存在 2 处契约不匹配, 表现为 `TypeError` 转 500:

### 3.1 `SchedulerEngine.get_history` — 不支持 start/end/status
**Before**:
```python
def get_history(self, job_id=None, limit=50, offset=0) -> List[dict]:
    # 只支持 job_id + 分页
```

**After** (`engines/scheduler_engine.py:482-528`):
```python
def get_history(self, job_id=None, limit=50, offset=0,
                start=None, end=None, status=None) -> List[dict]:
    """支持 start/end/status 过滤 — 动态构造 WHERE 子句"""
```

**修复原因**: R2 阶段 `scheduler_routes.py:233` 调用 `sched.get_history(job_id, limit, offset, start=..., end=..., status=...)`, 但引擎签名不接受这些 kwarg, 触发 `TypeError: got an unexpected keyword argument 'start'`。

### 3.2 `SchedulerEngine._build_trigger` — cron_expression 未翻译
**Before**:
```python
@staticmethod
def _build_trigger(job_def):
    if job_def.trigger_type == "cron":
        return CronTrigger(**job_def.trigger_config)  # cron_expression 不是 CronTrigger 字段
```

**After** (`engines/scheduler_engine.py:351-376`):
```python
@staticmethod
def _build_trigger(job_def):
    if job_def.trigger_type == "cron":
        cfg = dict(job_def.trigger_config or {})
        if "cron_expression" in cfg and isinstance(cfg["cron_expression"], str):
            # R2 阶段约定: trigger_config 中含 cron_expression 字符串
            # CronTrigger 期望: minute/hour/day/month/day_of_week 个体字段
            expr = cfg.pop("cron_expression").strip()
            parts = expr.split()
            if len(parts) != 5:
                raise ValueError(...)
            minute, hour, day, month, dow = parts
            cfg.setdefault("minute", None if minute == "*" else minute)
            # ... 其他 4 字段
        return CronTrigger(**cfg)
```

**修复原因**: R2 阶段 `CreateJobRequest.trigger_config` 允许 `{"cron_expression": "0 3 * * *"}` 形式 (由 `validate_trigger_config` 校验), 但 `CronTrigger` 期望个体字段 `{minute: 0, hour: 3, ...}` 形式。修复后两者契约对齐。

## 7. 关键决策与教训

### 7.1 验证器分层 (R2 已确定, R2.5 复用)

| 层 | 工具 | 触发 |
|----|------|------|
| **Field 层** | Pydantic `Field(..., min_length, max_length, ge, le, pattern, regex)` | 422 (Pydantic 自动) |
| **Model 层** | Pydantic `@model_validator(mode="after")` 中 `raise HTTPException(...)` | 400/422 (FastAPI) |
| **Handler 层** | `validate_xxx(value, name)` 函数首行调用 | 400 (显式 raise) |
| **Exception 层** | `try/except` 兜底 (保留 5xx, 不允许吞) | 500 (真正异常) |

### 7.2 model_validator 必 raise HTTPException (R2 经验)

`SchedulerHistoryParams` 的 `start > end` 校验在第一版用 `ValueError`, TestClient 下变 500; 改 `HTTPException(400)` 后正常 4xx。FastAPI `Depends(YourModel)` 不会把 `ValueError` 转 422, 必须显式 raise HTTPException。

### 7.3 SSRF 三层防御 (不可省略任一)

- 仅字面 hostname 黑名单 → 攻击者用 `attacker.com` 解析到 `127.0.0.1` 绕过
- 仅字面 IP 私网检查 → 攻击者用 hostname DNS 绕过
- 仅 DNS 解析后私网检查 → 攻击者用 IP 字面绕过

必须 1+2+3 三层。Pydantic v2 `field_validator` 内 `raise HTTPException` 在 webhook_routes 中被保留为 400 (不是 422), 但仍是 4xx 拒绝, 符合 R2 bad_params-200 → 4xx 规范。

### 7.4 引擎层 vs 路由层契约

R2 把验证器接到路由, 但**没有同步更新引擎签名**。`get_history` 接受 kwarg 但引擎不支持, `CronTrigger` 期望个体字段但路由传 `cron_expression` 字符串。R2.5 阶段补齐了这 2 处契约, 否则即使 Pydantic 校验通过, 真正执行也是 500。

教训: **路由层 + 验证层 + 引擎层 = 完整 3 段**, 只改前 2 段是 PARTIAL PASS。R2.5 任务就是补这最后一公里。

### 7.5 命名空间白名单 (task_id 7 种)

R1 `validate_id` 走 `^[a-zA-Z0-9_\-]{1,128}$` (通用), R2 `validate_task_id` 走 `^(task|job|batch|run|del|wh|mig)_[a-zA-Z0-9_]{4,64}$` (严格)。两套并存:
- 通用资源 (含 `preset_xxx`): R1 `validate_id`
- 异步任务 ID: R2 `validate_task_id`

`preset_xxx` 不在 task_id 白名单, 但走 R1 `validate_id` 仍合法, 详见 `tests/unit/test_r2_w4_validators.py::TestValidateTaskId::test_img_legacy_id_rejected`。
