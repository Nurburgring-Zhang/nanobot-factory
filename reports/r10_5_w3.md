### 2.2 99.9% 承诺要点

- **月停机上限**: 43 分钟 (按 30 天计)
- **不可用定义**: 5xx 错误率 ≥ 1% 持续 60s **或** /readyz 503 持续 60s **或**
  p95 超阈值持续 5min
- **不计入**: 客户端 4xx / 网络问题 / 计划维护 (48h 通知) / 不可抗力

### 2.3 RTO/RPO 矩阵

| 级别 | 场景 | RTO | RPO |
|---|---|---|---|
| P0 | 全站不可用 | **< 30 min** | **< 5 min** |
| P1 | 单服务降级 | < 2 hours | < 15 min |
| P2 | 边缘功能异常 | < 1 business day | < 1 hour |
| P3 | UI 小 bug | 下次发版 | N/A |

**支撑 RPO 5min 的备份策略**:
- SQLite `imdf.db`: `sqlite3 .backup` + WAL ship, 每 5 min
- OSS 对象: 三副本 + 跨区复制, 实时, RPO = 0
- 周日凌晨 03:00 自动 PITR 演练到独立 namespace (5 min 内完成)

### 2.4 容量规划 (3 档)

#### Tier A — 1 万 (10K) 数据集

- SQLite 单机 (WAL mode)
- 1-2 worker 副本, 1 CPU / 1Gi
- 本地 disk + 每日 OSS 备份
- **预期 p95 < 200ms**
- 月成本: $50 (云) / ¥800 (自建)
- SLO: 99.5% (开发) / 99.9% (生产)

#### Tier B — 10 万 (100K) 数据集 — **本 SLA 标准档**

- SQLite 主从 **或** PostgreSQL 单实例
- 3-5 worker 副本, 2 CPU / 2Gi
- 本地 NVMe + OSS 双副本
- **Redis 1GB** (LRU session / rate limit)
- **预期 p95 < 500ms**
- 月成本: $300 (云) / ¥4000 (自建)
- SLO: **99.9%**

#### Tier C — 100 万 (1M) 数据集

- **PostgreSQL 主从 + 读写分离 + pgbouncer**
- 8-16 worker 副本 (KEDA), 4 CPU / 4Gi
- OSS + CDN + MinIO
- **Redis 8GB cluster** (3m + 3r)
- **Elasticsearch 3 节点** (高级搜索)
- **RabbitMQ / Kafka** (任务调度)
- 预期 p95 < 1000ms
- 月成本: $2000 (云) / ¥25000 (自建)
- SLO: 99.95% (企业档, SLA addendum)
