## 二、修复详情

### 2.1 `model-manager.js` (14 → 104 行)

| 修复点 | 修改 |
|--------|------|
| **语法** | `mm_downloadModels` 闭合 `try/catch` 块; `mm_installGuide` / `mm_modelDetail` 独立函数声明, 各自占独立行 |
| **硬编码 5** | 改为从 `r?.cloud_vendors ?? cloud_count ?? vendors` 读真实值, 无值时显示 `-` |
| **硬编码网关 🟢** | 改为读 `r.gateway` / `gateway_status` / `gateway_ok` 字段, 分别映射 🟢/🟡/🔴/⚫ |
| **硬编码 "已安装"** | `mm_modelDetail` 从 `window.__mm_models_cache` 查找真实模型, 字段含 `installed`/`status`/`available` 三种命名, 动态渲染 |
| **installGuide 硬编码 pip 文本** | 改为 `GET /api/local-models/install-guide` 拉取 5 个模型的真实安装指南, 失败时回退到默认提示 |
| **XSS** | 所有用户输入 (id/name/path/user) 用 `sanitizeHTML` 包裹 |

### 2.2 `scheduler-center.js` (15 → 100 行)

| 函数 | 旧实现 | 新实现 |
|------|--------|--------|
| `sc_triggerAll` | `showToast('正在触发...')` | 1) 从 `window.__sc_jobs_cache` 拿当前 jobs 列表 2) 过滤掉 paused/disabled 3) `confirm()` 二次确认 4) `showGlobalLoading` + `Promise.allSettled` 并行 `POST /api/scheduler/jobs/{id}/run` 5) 统计 ok/fail, 显示结果 6) 500ms 后 re-render |
| `sc_filter` | `function sc_filter(v){}` | 读 `<input id="sc-search">` 值, 与 `data-name` 属性做 substring 匹配, 隐藏不匹配卡片, 显示"无匹配"提示 |
| `sc_filterByStatus` | `function sc_filterByStatus(v){}` | 读 `<select id="sc-status-sel">` 值, 与 `data-status` 属性做精确匹配, 与 sc_filter 的搜索值是 AND 关系 |
| `sc_runOne` | 已存在 | 加 `setTimeout(renderSchedulerCenter, 500)` 让 scheduler 把状态写入 history 后再渲染 |
| `sc_pauseOne` | 已存在 | 同上 |

**注**: 后端 scheduler_routes.py 没有 `trigger-all` 端点, 所以用前端串行/并行循环调单 job 触发。前后端契约遵循 R2-Worker-4 校验过的 `/api/scheduler/jobs/{id}/run`。

### 2.4 `audit-logs.js` (10 → 126 行)

| 函数 | 旧实现 | 新实现 |
|------|--------|--------|
| `al_refresh` | `function al_refresh(){location.reload()}` | 1) 把按钮置为 `⏳ 刷新中...` 视觉反馈 2) `await _al_loadAndRender()` 重调 `/api/v1/audit-logs?size=200` 3) 成功后 `showToast` 4) 失败 catch 后也 `showToast` 5) `finally` 恢复按钮 |
| `al_filter` | `function al_filter(v){}` | 1) 渲染时给每条 log 加 `data-search="<method+path+user+ts>.toLowerCase()"` + `data-level="info\|warn\|error"` 2) `al_filter()` 读 input+select, AND 匹配 3) level 由 `status_code` 自动分类: 5xx→error, 4xx→warn, 其他→info |
| `renderAuditLogs` | 直接 fetch | 抽到 `_al_loadAndRender`, `renderAuditLogs` 和 `al_refresh` 共用, 保证响应结构一致 |
| 字段映射 | 旧 `r?.logs \|\| r?.items` (错) | 新 `r?.data?.items \|\| r?.items \|\| r?.logs \|\| r?.data?.logs` (兼容 audit_routes.py 实际返回) |
| 系统状态卡 | 硬编码 `🟢` | 改为基于返回数据 5xx 错误数动态: 0=🟢, ≤5个 4xx=🟢, >5个 4xx=🟡, 5xx 出现=🔴 |

## 六、未做的事项 (留给验证者/后续)

1. **sc_triggerAll 性能**: 100+ 任务时 `Promise.allSettled` 并行可能触发 scheduler 限流, 当前实现是 1 次全发。后续 R4 可加分批/限流。
2. **sc_filter 后端过滤**: 当前是前端本地 substring 过滤。R2 阶段 scheduler 列表本身支持 status filter, 后续可加 server-side 分页+过滤。
3. **al_filter 时间范围**: 后端 audit_routes 支持 `dr.start/dr.end`, 但当前 UI 没暴露日期选择器。可在 R4 加 `<input type="date">`。
4. **mm_installGuide 降级文案**: 后端 5xx 时仍走硬编码 fallback, 提示"pip install sentence-transformers"。
