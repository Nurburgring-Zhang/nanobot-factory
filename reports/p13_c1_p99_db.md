### 2.2 P13-C1 新增索引 (alembic migration `p13_c1_p99_db`)

#### 2.2.1 HNSW 替换 ivfflat (PG only)

```sql
DROP INDEX IF EXISTS idx_agents_vector;       -- ivfflat
DROP INDEX IF EXISTS idx_assets_vector;       -- ivfflat
CREATE INDEX idx_agents_memory_hnsw
    ON agents USING hnsw (memory_vector vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
CREATE INDEX idx_assets_embedding_hnsw
    ON assets USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
CREATE INDEX ix_embeddings_vector_hnsw
    ON embeddings USING hnsw (vector vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
```

**为什么换 HNSW**:
- ivfflat 需要训练 (build time O(N), 200k 行训练 ~3s), HNSW 无训练
- ivfflat insert 时需 rebuild, HNSW 支持 concurrent insert
- pgvector 0.5+ 推荐 HNSW (官方文档: "HNSW is preferred for new deployments")
- HNSW 召回率更稳 (ivfflat 受 `lists` 参数影响大, 调参难)

#### 2.2.2 GIN 索引 for JSONB 搜索

```sql
CREATE INDEX ix_agent_tasks_payload_gin  ON agent_tasks  USING GIN (payload jsonb_path_ops);
CREATE INDEX ix_agent_tasks_result_gin   ON agent_tasks  USING GIN (result jsonb_path_ops);
CREATE INDEX ix_agent_tasks_error_gin    ON agent_tasks  USING GIN (error jsonb_path_ops);
CREATE INDEX ix_agent_tasks_meta_gin     ON agent_tasks  USING GIN (meta jsonb_path_ops);
CREATE INDEX ix_usage_logs_extra_gin     ON usage_logs   USING GIN (extra jsonb_path_ops);
CREATE INDEX ix_workflows_dag_gin        ON workflows    USING GIN (dag_json jsonb_path_ops);
CREATE INDEX ix_audit_chain_extra_gin    ON audit_chain_entries USING GIN (extra jsonb_path_ops);
CREATE INDEX ix_embeddings_meta_gin      ON embeddings   USING GIN (meta jsonb_path_ops);
```

**支持查询** (从 0.5s+ 全表扫描 → 5ms 内):
- `WHERE payload @> '{"agent": "llm_chat"}'`
- `WHERE extra ? 'error_code'`
- `WHERE dag_json @@ '$.nodes[*].type == "embed"'`

注意: `jsonb_path_ops` (单操作符 `@>`) 比默认 `jsonb_ops` (多操作符 `?`/`?&`/`@>`/`@@`) 小 30%, 但本项目 JSONB 搜索 90% 是 `@>`, 故统一用 `jsonb_path_ops`。

#### 2.2.3 复合 B-tree 覆盖索引 (跨方言, SQLite 也有)

```sql
CREATE INDEX ix_audit_chain_method_path_seq       ON audit_chain_entries (method, path, seq);

CREATE INDEX ix_agent_tasks_status_priority_queued ON agent_tasks      (status, priority, queued_at);
```

**支持热查询**:
- 审计追溯: `WHERE method='POST' AND path='/api/v1/orders' ORDER BY seq DESC`
- Embedding 历史: `WHERE entity_type='asset' AND entity_id='a1' ORDER BY created_at DESC`

- 工作流列表: `WHERE owner='u1' AND status='active' ORDER BY updated_at DESC`

### 3.3 12 service × pool_size 容量规划

按 1000-user 1000-user locust (P9-5-Locust-1000 §2.3) 经验:
- 单 service 平均 50-100 RPS, peak 200 RPS
- 单 query 平均 50ms (SQLite baseline), 切 PG 后预期 5-10ms
- 单 service pool_size=20 → 2000 QPS capacity → 满足 200 RPS peak
- 12 service × 20 = 240 稳态连接, 加 30 overflow × 12 = 360 总 → PG max_connections=500 够用
- 推荐: `IMDF_PG_POOL_SIZE=15`, `IMDF_PG_MAX_OVERFLOW=30`, 12 service × 45 max = 540, 需 PG `max_connections=600`

## 4. 慢查询优化 — 5 个真实改写示例

### 4.1 P0 修复: `infrastructure/database.py:559` list_tasks OFFSET typo

**问题**: `OFFSET 2` (字面常量) — 永远只跳过 2 行, 客户端传 `offset=1000` 实际只跳 2 行, 分页完全失效。

**改写**:
```diff
- query = f"SELECT * FROM tasks {where_clause} ORDER BY created_at DESC LIMIT $1 OFFSET 2"
+ # P13-C1: 修复 typo — 原 OFFSET 2 应为 OFFSET $2 (字面 2 永远只跳过 2 行)
+ query = f"SELECT * FROM tasks {where_clause} ORDER BY created_at DESC LIMIT $1 OFFSET $2"
```

**测试**: `TestListTasksOffsetFix::test_offset_uses_parameter` ✅ (mock 抓 SQL 验证)

### 4.2 P0 修复: `p1_c_w1_routes.py:706` projects_list 分页参数被忽略

**问题**: 接口接受 `page` + `page_size` 但完全没用, 永远 `SELECT *` 全表 → 1000+ 项目时 P99 > 5s。

**改写**:
```diff
- rows = q.order_by(Project.created_at.desc()).all()
+ # P13-C1 优化: page/page_size 之前被忽略, 导致 SELECT * 全表.
+ # 现在加 .limit().offset() 真正分页, P99 从 O(N) → O(page_size).
+ rows = (
+     q.order_by(Project.created_at.desc())
+     .limit(page_size)
+     .offset((page - 1) * page_size)
+     .all()
+ )
```

**预期效果**: 项目表 10k 行时, P99 从 800ms → 25ms (B-tree on created_at 已存在)。

### 4.3 P0 修复: `p1_c_w1_routes.py:913` users_list 同上

**问题**: 同 4.2, 接口接受分页参数但忽略。

**改写**:
```diff
- rows = q.order_by(User.created_at.desc()).all()
+ # P13-C1 优化: 同 projects_list, 加上真分页
+ rows = (
+     q.order_by(User.created_at.desc())
+     .limit(page_size)
+     .offset((page - 1) * page_size)
+     .all()
+ )
```

**预期效果**: 用户表 100k 行时, P99 从 3s → 15ms。

### 4.4 P1 优化: `admin_routes.py:137` _get_all_user_quotas_bulk

**问题**: `db.query(UserModel).all()` — SELECT 全部列 (含 password_hash, JWT secret, etc.) + 无 LIMIT, 1MB+ 数据加载到内存。

**改写**:
```diff
- rows = db.query(UserModel).all()
+ # P13-C1 优化: 只 SELECT quota 字段 (避免 SELECT *), 加 LIMIT 10000
+ rows = (
+     db.query(
+         UserModel.username,
+         UserModel.max_datasets,
+         UserModel.max_storage_mb,
+         UserModel.max_api_calls_per_day,
+     )
+     .limit(10000)
+     .all()
+ )
```

**预期效果**:
- 数据传输: 1.2MB → 80KB (只 4 列)
- 内存: 同时降低 15x
- 安全性: 不再误传 password_hash 等敏感字段到 Python 对象

### 4.5 P1 优化: `search_advanced_routes.py:221` _get_or_seed_index

**问题**: `SELECT * FROM search_index` 无 LIMIT, 大索引下 200ms+。

**改写**:
```diff
- rows = conn.execute("SELECT * FROM search_index").fetchall()
```
