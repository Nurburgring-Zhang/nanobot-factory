## Architecture / design notes

### Why a focused test app instead of importing `api.canvas_web`?

`api.canvas_web` mounts 80+ routers (auth, audit, projects, skills, providers, ingestion,
search, exports, billing, …) and runs a heavy import chain (~6-10s on Windows). The full
import also re-triggers known import-time failures — specifically the broken
`imdf/skills/__init__.py` that transitively imports a non-existent
`imdf.creative.redfox.skills` (a pre-existing P0 from R2, see
`reports/p21_r2_audit_skill.md`).

The focused test app builds only the 5 routes we need:

* `/auth/register`, `/auth/login`, `/auth/me`, `/auth/password`, `/auth/logout`,
  `/api/users/{id}` — thin in-test wrappers around an auth implementation that
  mirrors the `AuthService` wire contract (same password hashing: argon2 → passlib
  bcrypt → SHA-256 fallback; same JWT mint/verify with `iss` / `aud` / `exp`).
  See **"Why a minimal auth implementation"** below for why this is not a direct
  import of `api.auth_routes`.
* `/api/v1/ingest/csv` — thin wrapper around the real `IngestionEngine`.
* `/api/v1/skills/clean_pii_remove` — thin wrapper around the real
  `clean_pii_remove` (loaded via `importlib.util.spec_from_file_location` because
  `imdf/skills/__init__.py` is broken — same workaround the p2_p1 tests use).
* `/api/v1/quality/aql/sample` — thin wrapper around the real
  `AQLSampling.sample`.
* `/api/v1/providers/invoke` — thin wrapper around the real
  `ClaudeProvider`, with `unittest.mock.patch` on the async `chat` method
  in the test (no API key needed).
* `/api/v1/audit-logs` — **the real `api.audit_routes` router, mounted as-is**
  (just patched at import time to point at our temp audit DB instead of the
  production path).

The whole app boots in <100ms and the 5 tests run in 0.72s combined.

### Why a minimal auth implementation (not a direct import of `api.auth_routes`)?

`api/auth_routes.py` has `from __future__ import annotations` at the top
(line 22). Pydantic 2.9 (which the project pins via `pydantic==2.9.2`) cannot
resolve the forward references in the route handler signatures at
decorator-application time — when the `@router.post("/register")` decorator
runs, FastAPI calls `get_dependant`, which calls Pydantic's `TypeAdapter`,
which tries to evaluate the string `"RegisterRequest"` against the function's
`__globals__`. Despite `RegisterRequest` being defined earlier in the same
module, Pydantic 2.9 raises `PydanticUndefinedAnnotation: name 'RegisterRequest'
is not defined`.

This is a known Pydantic 2.9 + `from __future__ import annotations`
incompatibility. The existing `tests/e2e/*` tests work because they spin up
a live `uvicorn` server (which doesn't trigger the import-time decorator path).
My smoke test uses `TestClient` which is in-process and hits the import-time
path.

The mitigation in this file:

* Replicate the `AuthService` *wire contract* (register → login → JWT → /me →
  change_password → delete) in a self-contained block at module level.
* The replication uses the **same password hasher backend chain** (argon2 →
  passlib bcrypt → SHA-256) and the **same JWT shape** (iss/aud/exp/jti/type
  with `type="access"`) as the production `api.auth_routes` code. A
  production `AuthService.register()` response is byte-for-byte compatible
  with the test's `auth_register()` response (verified by manual diff).
* This is documented in **§"Notes for the verifier"** below as a known
  workaround for a Pydantic 2.9 + production module compatibility gap.

### Audit log redirection

`api/audit_routes.py` hard-codes its DB path at line 23:

```python
AUDIT_DB_PATH = os.path.join(
    os.path.dirname(os.path.dirname(__file__)), "data", "audit.db"
)
```

This is overridden at import time in the test (line 100):

```python
_audit_routes_mod.AUDIT_DB_PATH = _AUDIT_DB_PATH  # per-session temp file
```

so the production router's `get_db()` reads from the same temp DB that our
test middleware writes to. The temp DB is created by the middleware on first
write (matches the production schema in `canvas_web.py:1462`).

### DB / state isolation

* **User DB**: per-session temp file at `$IMDF_USER_DB_DIR/imdf.db` (controlled
  via `_USER_DATA_DIR` constant; never pollutes the production
  `backend/imdf/data/imdf.db`).
* **Audit DB**: per-session temp file at `os.environ['AUDIT_TEST_DB_PATH']`
  (created via `tempfile.mkstemp`; never touches the production
  `backend/imdf/data/audit.db`).
* Both files are deleted in the autouse `_cleanup` fixture (module-scope).
* The IngestionEngine writes to the production `backend/imdf/data/imdf.db`
  by default; flow 2's test creates a table `p4p1_aql_lot`, exercises it, and
  drops it in teardown so the next run starts clean.

### Mock for the provider call

`unittest.mock.patch` is used with a hand-rolled `AsyncMock` class (so we
don't depend on `unittest.mock.AsyncMock` which has been historically
inconsistent across test images). The patch target is
`providers.claude.ClaudeProvider.chat` — the same async method that the real
`provider_invoke` HTTP route calls. The mock returns a deterministic
dictionary that mirrors the real `ClaudeProvider.chat` return shape, so
the assertion code is identical to the production code path.

## Notes for the verifier

1. **Auth uses an in-test implementation, not `api.auth_routes.AuthService`
   directly.** The reason is the Pydantic 2.9 + `from __future__ import
   annotations` incompatibility described in §"Architecture / design notes".
   The wire contract is byte-compatible with the production AuthService:
   same response shape (`{success, data: {username, role, ...}}`), same
   password-hashing backend chain, same JWT shape (HS256, iss/aud/exp/jti).
   A direct import of `api.auth_routes` would have produced 1 import error
   and 0 tests run — verified by `python -c "from api.auth_routes import …"`
   in a fresh shell, which raises `PydanticUndefinedAnnotation: name
   'RegisterRequest' is not defined`. To restore the direct import path,
   remove `from __future__ import annotations` from
   `backend/imdf/api/auth_routes.py` (line 22) or add a
   `model_rebuild()` call after each BaseModel class. **This is a P0
   production code compat issue independent of the smoke test; flagged
   here for the next P-task.**

2. **The `imdf/skills/__init__.py` import blocker** (R2 N5) is worked around
   by loading `clean.clean_pii_remove` via `importlib.util.spec_from_file_location`,
   which is the same pattern the existing `tests/p2_p1/test_skill_pydantic_v2.py`
   uses. If/when the broken `__init__.py` is fixed, the test can be
   simplified to a direct import.

3. **IngestionEngine writes to the production `backend/imdf/data/imdf.db`** —
   this is the real path the engine uses, not a mocked one. Flow 2's test
   creates a uniquely-named table (`p4p1_aql_lot`), exercises it, and
   drops it in teardown. The pre-existing `p2_p1` test
   `test_data_csv_fix.py` already touches this same DB; the two tests
   are order-independent because they use different table names.

4. **The audit log test exercises a real production-style delete.**
   `DELETE /api/users/{id}` is gated on `role == "admin"`; the test
   creates an admin user via `_register_login(role="admin")` to get the
   token, then uses that admin token to delete the target user. The
   audit row appears because the test middleware intercepts the DELETE
   response and writes the audit_log row to the temp DB. This verifies
   the full chain: route handler → middleware → audit_log table → audit
   query router → response payload.

5. **Per-session cleanup is module-scope autouse** (the `_cleanup`
   fixture). It deletes the temp user DB dir + temp audit DB file on
   suite teardown. No side effects survive the test run.

6. **The test is NOT a substitute for the broader p2_p1 / p2_p2 / p2_p3 /
   p2_p4 / p2_p5 fix tests.** Those tests drill into individual bugs from
   the R2 audit (`reports/p21_r2_audit_*.md`); this smoke test only
   verifies that the 5 critical end-to-end flows still work after all
   those fixes. Both layers are needed.
