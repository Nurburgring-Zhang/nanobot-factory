## 二、logging_setup.py 设计要点

- **structlog 替代 print**:集中入口 `configure_logging()` / `get_logger(name)`,幂等
- **三路输出**:stderr (console) + `logs/access.log` (RotatingFileHandler) + `logs/error.log`
- **JSON 输出**:结构化 kwargs → JSON 字段,适合 ELK / Loki / Datadog 摄取
- **ContextVar 跨 await**:`_trace_id_var` / `_request_id_var` 不用显式 threading
- **API**:`set_trace_id()` / `get_trace_id()` / `set_request_id()` / `clear_trace_context()`

## 三、middleware.py 设计要点

- **TraceIDMiddleware**:
  - 读 `X-Trace-Id` header,缺失则生成 UUID4 hex
  - 绑定到 ContextVar,所有 structlog 事件自动带 trace_id
  - 响应回写 `X-Trace-Id` + `X-Request-Id`
- **RequestLoggingMiddleware**:
  - 每个请求发一条结构化 "request completed" 事件 (method/path/status/elapsed/trace_id)
  - 慢请求 >1s 触发 WARNING
  - 转发到 `engines.metrics.record_request` 上报 Prometheus
- **path normalization**:
  - UUID 替换 `{id}`,长数字替换 `{n}` → cardinality 不爆
  - 暴露给 metrics,避免每次路径里具体 ID 都创建新 series

## 四、healthz.py / readyz.py 设计要点

- **/healthz** (liveness):
  - 200 OK 即可,无依赖检查 — K8s 频繁轮询
  - 返回:status / service / version / uptime / timestamp / pid
  - 用途:`livenessProbe`,只在进程死掉时重启
- **/readyz** (readiness):
  - 200 仅当所有 critical check 通过 (DB + 磁盘)
  - 503 任何 critical check 失败 → K8s 停止路由流量到此 pod
  - SQLite ping `data/imdf.db`,磁盘可用 >200MB 才算 OK
  - 与 `/api/v1/health/ready` 行为一致,但顶层路径独立
